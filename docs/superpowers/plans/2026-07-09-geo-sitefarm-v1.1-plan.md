# GEO 计费系统 V1.1 站群模块实施计划

> **项目**：GEO 计费系统  
> **版本**：V1.1（在 V1.0 基础上新增「站群模块」）  
> **策略**：方案二 —— 以站群模块为新基座起点，沉淀统一设计系统，再逐步迁移 V1.0  
> **日期**：2026-07-09  
> **适用对象**：产品经理、前端开发、AI 协作者

---

## 一、目标与范围

### 1.1 核心目标
在现有 `geo-landing`（Next.js + React 19 + TypeScript + Tailwind 4 + shadcn/ui）项目基础上，新增「站群模块」，覆盖 **超级管理员端、SAAS 租户端、终端用户端** 三个端口，形成可演示、可后续迭代的前端实现。

### 1.2 本次范围

| 端口 | 新增模块/页面 |
|------|---------------|
| 超级管理员端 | 站群权限配置（SAAS 用户管理）、网站模板管理、TDK 变量管理 |
| SAAS 租户端 | 基础资料管理、网站模板管理、训练词管理、企业图片/知识库 |
| 终端用户端 | 基本信息维护、生成站点查看 |

### 1.3 不在本次范围
- V1.0 已有页面（ dashboard、客户管理、订单管理等）本次不迁移；
- 后端接口联调以 Mock 数据为主，预留接口类型；
- 真正的站点批量生成引擎（后端逻辑）本次仅做前端配置与展示。

---

## 二、总体策略

采用「先站稳群、再迁 V1.0」的渐进式策略：

1. **不搬 V1.0**：先集中资源把站群模块做扎实；
2. **沉淀组件**：把站群里高频出现的表单、表格、上传、城市选择、模板预览等抽象成可复用组件；
3. **统一规范**：颜色、字体、间距、按钮、表单样式全部对齐现有 `geo-landing` 设计系统；
4. **逐步迁移**：站群模块完成后，按业务优先级把 V1.0 页面迁过来。

### 阶段 6：V1.0 页面逐步迁移（后续按需排期）

**目标**：把 V1.0 中尚未迁移到 `geo-landing` 的页面，按优先级逐步迁移到新的设计系统。

#### 迁移优先级建议
1. **高优先级**：登录页、工作台、客户管理、订单管理（使用频率高）；
2. **中优先级**：财务管理、员工管理、数据报表；
3. **低优先级**：系统设置、操作日志、消息公告。

**产出物**
- 每迁移一个模块，复用阶段 1 沉淀的组件；
- 更新 `PageRouter`、`PORT_CONFIG`、`PAGE_TITLES`。

**验收标准**
- 迁移后的页面与站群模块风格一致；
- 不破坏原有功能；
- 逐步减少 V1.0 旧资产（Axure/HTML 原型）的依赖。

## 五、关键决策点

| 决策 | 建议 | 说明 |
|------|------|------|
| 是否复用现有 `DataTable`？ | 是 | 现有表格组件已满足基本需求，仅做少量扩展 |
| 模板预览用 Iframe 还是图片？ | 先 Iframe | 更能体现真实效果，后续可切图片快照 |
| TDK 变量一期是否写死？ | 建议保留超管配置入口 | 即使一期变量少，也保留扩展性 |
| 站群菜单是否独立一个分组？ | 是 | 在三个端口都新增「站群管理」或「快通站群」分组 |
| 城市选择数据从哪里来？ | 先用静态城市列表 | 后续可接行政区划接口 |

## 六、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 站群模块页面多、字段多，开发周期拉长 | 中 | 按阶段拆分，每个阶段独立可验收 |
| 模板工作台（图文编辑）复杂度超预期 | 高 | 先实现「预览模式 + 简单变量替换」，复杂编辑后续迭代 |
| 三端数据不统一 | 中 | 阶段 0 先定义统一类型和 Mock 数据 |
| V1.0 迁移被无限期推迟 | 中 | 阶段 6 必须明确优先级和时间节点 |
| AI 生成代码质量不稳定 | 中 | 每个阶段结束后人工走查，关键组件手写 |
